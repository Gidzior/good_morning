## Phase 1: Helper `apiFetch` + migracja silent-fail mutations

**1a. Utworz `frontend/src/lib/api.ts`:**
```ts
// Jedyny punkt fetchowania API z rzucaniem bledow: !res.ok → Error z trescia
// {error} z serwera gdy dostepna, inaczej "HTTP <status>".
export async function apiFetch<T>(url: string, init?: RequestInit): Promise<T> {
  const res = await fetch(url, init);
  if (!res.ok) {
    let detail = '';
    try {
      const body: unknown = await res.json();
      if (typeof body === 'object' && body !== null && 'error' in body && typeof (body as { error: unknown }).error === 'string') {
        detail = (body as { error: string }).error;
      }
    } catch {
      // body nie-JSON — zostaje sam status
    }
    throw new Error(detail || `HTTP ${res.status}`);
  }
  return (await res.json()) as T;
}
```
Uwagi: strict TS, `unknown` + narrow (zero `any`). Catch przy parsowaniu error-body jest CELOWY fallback (komentarz w kodzie) — blad i tak jest rzucany.

**1b. Migruj 9 handlerow.** Schemat kazdego: wrap w try/catch → `apiFetch` → sukces jak dotychczas; catch → `console.error` + lokalny stan bledu widoczny w UI (istniejacy wzorzec `ErrorMsg` / inline tekst, ZERO nowych bibliotek). Stan bledu czyszczony przy kolejnej probie.

- **Weather.tsx addCity/removeCity (:110-131):** dodac `actionError: string | null`; pokazac w modalu ustawien miast (inline `<div className="error-msg">`), czyszczone na nowa probe.
- **RSS.tsx handleAddFeed (:70-84):** w catch dodatkowo `setAdding(false)` (fix wiszacego adding); error inline w settings modal. handleRemoveFeed (:86-90) analogicznie.
- **App.tsx addRssWidget/addTodoList (:84-104):** dialog zostaje otwarty, error inline w dialogu (`dialogError: string | null`, czyszczone przy open/zmianie nazwy). deleteTodoList (:106-110): error pokazac tam gdzie wywolanie (sidebar/confirm) — najprosciej wspolny `actionError` w App renderowany jako `ErrorMsg` przy dialogu potwierdzenia.
